### (c) Response helpers (utils.js) — CONFIRMED
- sendErrorResponse(res, statusCode, message) -> res.status(code).json({ statusCode, message }).
  SINGULAR `message` for ALL errors.
- sendSuccessResponse(res, data, message=null) -> message ? { message, ...data } : data.
  i.e. it SPREADS the `data` arg at top level with `message` prepended. Invoice/cancel pass
  { statusCode:200, data:{...} } as the data arg, yielding { message, statusCode, data }.
  Register/update DO NOT use the helper — they build the identical { statusCode, message, data }
  shape with res.json directly.

### ENVELOPE-KEY QUIRK — MAJOR DRIFT from plan (most important finding)
- The plan generalizes "message (singular, writes/errors) vs messages (plural, reads)". On the FULL master
  surface this is WRONG. The TRUTH:
  * EVERY endpoint OUTSIDE the V2 membership sub-resource uses PLURAL `messages` for reads, writes, AND
    errors (helpers returnJSON / sendcursor all emit `messages`; handlers hand-build `messages`).
  * ONLY the 7 V2 membership handlers in memberships.js keep the singular `message`: writes
    (create/update/invoice/cancel) success bodies + ALL their 400/404/validation errors use SINGULAR
    `message`; their 3 reads (tiers list, members list, member detail) use PLURAL `messages`.
  * The 2 LEGACY membership reads (membership.js) use PLURAL `messages`.
  => Per-page rule for authoring: default to `messages` (plural) EVERYWHERE; use singular `message` ONLY on
     the 4 membership WRITE pages (create/update/invoice/cancel) — and even there the read siblings stay plural.
- checkJwt auth failures (utils.js): 401 {statusCode:401, messages:"Unauthorized"} (no/!bad token);
  404 {messages:"User in JWT not found"}; POST without write scope -> 403 {messages:"API KEY Scope not
  allowed to write"}. ALL PLURAL `messages`.
